## Поддерживаемые хранилища

### LocalStorage

Синхронное строковое хранилище. Блокирует главный поток на время операции.

- Квота: ~5–10 МБ (зависит от браузера)
- Недоступно в приватном режиме и по `file://`
- Данные сохраняются до явного удаления

### IndexedDB

Асинхронная транзакционная база данных в браузере.

- Квота: большая (зависит от браузера и свободного места)
- Может быть очищена браузером при нехватке места
- Ограничения в Safari/iOS, может блокироваться в приватном режиме

### Cache API

Асинхронное хранилище для запросов/ответов (Service Worker Cache).

- Требует HTTPS или `localhost`
- Недоступно по `file://`
- Доступность зависит от браузера

## Операции

Можно выбрать любой набор операций; все они выполняются последовательно для каждого хранилища.

| Операция    | Описание                                        |
|-------------|-------------------------------------------------|
| **Write**   | Записывает N ключей с заданными данными         |
| **Read**    | Читает те же N ключей                           |
| **Delete**  | Удаляет N ключей                                |
| **Iterate** | Перебирает все ключи с префиксом `bench-key-`   |

Для каждой операции измеряется суммарное время в миллисекундах и вычисляется **ops/sec** = количество ключей / время(с).

### Тип данных

#### JSON Objects

Данные представляют собой JSON-массив объектов. Пользователь может задать шаблон, который автоматически дублируется до нужного размера.

Шаблон по умолчанию:
```json
[
  {
    "id": 1,
    "text": "Example",
    "value": 42,
    "tags": ["alpha", "beta"]
  }
]
```

Правила дублирования:
- Если размер одного элемента меньше целевого, массив заполняется копиями объекта.
- Поле `id` автоматически инкрементируется для каждого нового экземпляра.

#### Raw Bytes

Бинарные данные в виде `Uint8Array`, закодированные в base64 для хранилищ с текстовым форматом.

Доступные паттерны:

| Паттерн    | Описание                                        |
|------------|-------------------------------------------------|
| **random** | Криптографически случайные байты (`crypto.getRandomValues`) |
| **zeros**  | Все байты равны `0x00`                          |
| **repeat** | Повторяющаяся последовательность `0xAB 0xCD`    |

### Количество ключей

| Значение | Примечание                                    |
|----------|-----------------------------------------------|
| 1        | Для проверки латентности одиночной операции   |
| 100      | Стандартный сценарий                          |
| 10 000   | Нагрузочный сценарий; операции идут пакетами по 500 ключей с паузой между батчами, чтобы не блокировать интерфейс |

## Интерфейс

Приложение разбито на три вкладки.

### Конфигурация

Выбор всех параметров теста:
- Хранилища: можно включить любое сочетание LocalStorage / IndexedDB / Cache API.
- Операции: можно включить любое сочетание Write / Read / Delete / Iterate.
- Размер данных, тип данных и количество ключей.

Кнопка **«Сохранить и продолжить»** переходит на вкладку Run.

### Run

Запуск теста:
- Кнопка **▶ Запустить** запускает бенчмарк.
- Прогресс-бар показывает текущий шаг из общего числа шагов.
- Кнопка **■ Остановить** прерывает выполнение.
- В нижней части отображаются последние ошибки (до 5 последних из максимум 200).

Фазы выполнения:
1. **Cleanup** — очистка данных от предыдущих запусков.
2. **Availability check** — проверка доступности каждого хранилища.
3. **Warm-up** — одиночная запись/чтение для инициализации хранилищ.
4. **Run** — основные измерения.
5. **Export** — отправка результатов в Google Sheets.

После завершения вкладка автоматически переключается на **Results**.

### Results

Визуализация результатов в трёх режимах.

## Оговорки методологии

Цифры стоит читать с учётом нескольких принципиальных особенностей сравнения. Это не баги, а следствия разной природы API — они влияют на интерпретацию результатов.

### Raw Bytes + LocalStorage: равный объём в хранилище, но меньше полезных данных

LocalStorage хранит только строки в кодировке UTF-16 (2 байта на символ), поэтому бинарные данные приходится кодировать в base64 (+33% символов). Чтобы итоговый объём в хранилище совпадал с выбранным размером, исходный массив байт уменьшается в **8/3 ≈ 2.68** раза (`4/3` от base64 × `2` от UTF-16).

В результате при равном **footprint** (занятом месте в хранилище) LocalStorage переносит примерно **3/8 полезных байт** по сравнению с IndexedDB и Cache API, которые хранят сырые байты как есть. Сравнение выровнено по занимаемому месту, а не по количеству полезной информации — иначе LocalStorage может казаться «эффективнее» на чтении/записи.

### Read: что именно измеряется

Во всех трёх хранилищах операция Read измеряет получение **готовых к использованию данных** в память внутри замеряемого участка:

- **LocalStorage** — `getItem` возвращает строку целиком;
- **IndexedDB** — `get` возвращает значение после structured-clone десериализации;
- **Cache API** — `match()` возвращает только `Response` без чтения тела, поэтому дополнительно вызывается `arrayBuffer()`, иначе данные с диска не поднимаются в память и чтение выглядело бы искусственно быстрым.

### Разные модели конкурентности при записи

Запись пакета ключей выполняется в «естественной» для каждого API манере, и модели отличаются:

- **LocalStorage** — синхронный цикл `setItem` (блокирует главный поток);
- **IndexedDB** — все ключи пакета в одной транзакции;
- **Cache API** — параллельные `put` через `Promise.all`.

Разница в параллелизме влияет на абсолютные цифры записи и не приводится к единой модели.

## Технические детали

- **BATCH_SIZE** = 500 — размер пакета при `keyCount = 10 000`.
- Base64-кодирование разбито на чанки по 32 768 байт во избежание переполнения стека.
- Генерация случайных байт — чанки по 65 536 байт через `crypto.getRandomValues`.
- Ключи именуются по схеме `bench-key-{index}`.
- IndexedDB: база `web-storage-bench`, хранилище `records`, keyPath: `key`.
- Cache API: кеш `bench-v1`, ключи преобразуются в URL-вид `https://bench/{key}`.
- Стек: React 19, Vite, Recharts, Tailwind CSS 4.
